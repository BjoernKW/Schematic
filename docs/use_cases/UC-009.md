# Use Case: Configure UI Mount Path

## Overview

**Use Case ID:** UC-009
**Use Case Name:** Configure UI Mount Path
**Primary Actor:** Developer
**Goal:** Configure the URL path where the Schematic UI is mounted so that it does not conflict with the host application's existing routes
**Status:** Done

## Preconditions

- Schematic is added as a Maven dependency in the host application (UC-001 completed)
- Developer has write access to the host application's `application.yml` (or equivalent configuration file)
- Developer has identified a path conflict or wishes to use a non-default mount point

## Main Success Scenario

1. Developer opens the host application's `application.yml`.
2. Developer sets `schematic.path` to the desired URL path segment.
3. Developer saves the configuration file.
4. Developer restarts the application.
5. System picks up the new path value from configuration at startup and mounts the Schematic UI at `/<schematic.path>/tables`.
6. Developer navigates to the new path in a browser and confirms the UI is accessible.

## Alternative Flows

### A1: Configured Path Conflicts with an Existing Route

**Trigger:** The chosen path segment matches a route already registered by the host application (step 5)
**Flow:**

1. The host application's existing route takes precedence, and the Schematic UI may be inaccessible or partially shadowed.
2. Developer chooses a different path segment and repeats from step 2.

### A2: Configuration Property Omitted

**Trigger:** `schematic.path` is not present in `application.yml` (step 5)
**Flow:**

1. System uses the default path value `schematic` and mounts the UI at `/schematic/tables`.
2. Use case ends.

### A3: Root Path Also Configured

**Trigger:** Developer additionally sets `schematic.root-path` to match the host application's context path (step 2)
**Flow:**

1. Developer sets both `schematic.path` and `schematic.root-path` in `application.yml`.
2. Use case continues at step 3.
3. System mounts the UI relative to the configured root path at startup.

### A4: Application Fails to Start After Configuration Change

**Trigger:** An invalid or malformed value for `schematic.path` prevents the application from starting (step 5)
**Flow:**

1. System logs a startup error describing the invalid configuration value.
2. Developer corrects the value in `application.yml` and restarts.
3. Use case continues at step 5.

## Postconditions

### Success Postconditions

- The Schematic UI is accessible at the newly configured path
- All internal links within the Schematic UI resolve correctly relative to the new path
- No existing routes of the host application are affected

### Failure Postconditions

- The Schematic UI is either inaccessible or accessible only at the default path
- The host application's existing routes remain unaffected
- System logs a startup error if the configuration value is invalid

## Business Rules

### BR-001: Configuration Only

The mount path is set exclusively via `application.yml` (or equivalent Spring Boot configuration source); no code changes or annotations in the host application are required.

### BR-002: Default Path

When `schematic.path` is not configured, the UI is mounted at the default path `schematic`, making it accessible at `/schematic/tables`.

### BR-003: Restart Required

A path change takes effect only after the application is restarted; live reload of this property is not supported.

### BR-004: No Route Validation at Startup

Schematic does not validate whether the configured path conflicts with other routes in the host application; the developer is responsible for choosing a non-conflicting path.
