# Use Case: Truncate Table

## Overview

**Use Case ID:** UC-006
**Use Case Name:** Truncate Table
**Primary Actor:** Operator
**Goal:** Clear all rows from a table while preserving its structure, after explicitly confirming the intent
**Status:** Draft

## Preconditions

- Schematic is embedded in the running Spring Boot application (UC-001 completed)
- The application's datasource is connected to a reachable database
- Operator has access to the Schematic UI URL in a browser
- The target table exists in the database
- The datasource user has TRUNCATE (or equivalent DELETE) privilege on the target table

## Main Success Scenario

1. Operator navigates to the table list in the Schematic UI.
2. System displays the list of tables.
3. Operator selects the Truncate Table action for the target table.
4. System displays a confirmation dialog naming the table to be truncated and warning that all rows will be permanently deleted.
5. Operator confirms the truncate operation.
6. System removes all rows from the table.
7. System shows a success message confirming the table has been truncated.

## Alternative Flows

### A1: Operator Cancels Confirmation

**Trigger:** Operator dismisses or cancels the confirmation dialog (step 5)
**Flow:**

1. System closes the confirmation dialog without executing any operation.
2. Table data remains unchanged.
3. Use case ends.

### A2: Truncate Fails Due to Foreign Key Constraint

**Trigger:** Another table holds a foreign key reference to a row in the target table (step 6)
**Flow:**

1. System displays an error message indicating the table cannot be truncated because of an existing foreign key dependency.
2. Operator reviews the dependency and resolves it (e.g. truncating the referencing table first) before retrying.
3. Use case ends.

### A3: Truncate Fails Due to Insufficient Privileges

**Trigger:** The datasource user does not have the required privilege to truncate the target table (step 6)
**Flow:**

1. System displays an error message indicating insufficient database privileges.
2. Operator contacts a database administrator to obtain the required privilege.
3. Use case ends.

### A4: Database Connection Lost During Truncate

**Trigger:** The datasource cannot reach the database when the truncate is executed (step 6)
**Flow:**

1. System displays an error message indicating the database is unreachable.
2. Table row state is indeterminate; operator verifies the table contents after connectivity is restored.
3. Use case ends.

### A5: Table No Longer Exists at Execution Time

**Trigger:** The target table has been removed by another process between step 3 and step 6
**Flow:**

1. System displays an error message indicating the table was not found.
2. System refreshes the table list to reflect the current database state.
3. Use case ends.

## Postconditions

### Success Postconditions

- All rows in the target table are permanently removed
- The table structure (columns, constraints, indexes) remains intact
- The table continues to appear in the Schematic table list with a row count of zero

### Failure Postconditions

- The table's data remains unchanged
- System displays an error message indicating the cause of failure
- No other tables or data are affected

## Business Rules

### BR-001: Mandatory Confirmation

Every Truncate Table operation must display an explicit confirmation dialog naming the target table before executing; the operation cannot proceed without operator confirmation.

### BR-002: Data Loss Warning

The confirmation dialog must clearly communicate that all rows in the table will be permanently deleted and that this action cannot be undone.

### BR-003: Structure Preserved

Truncate removes only rows; the table definition, columns, indexes, and constraints are not affected.

### BR-004: Datasource Permissions Apply

The truncate executes under the privileges of the host application's configured datasource user; Schematic does not elevate or bypass database permissions.
