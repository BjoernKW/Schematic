# Use Case: Drop Table

## Overview

**Use Case ID:** UC-005
**Use Case Name:** Drop Table
**Primary Actor:** Operator
**Goal:** Permanently remove an unwanted table from the database after explicitly confirming the intent
**Status:** Draft

## Preconditions

- Schematic is embedded in the running Spring Boot application (UC-001 completed)
- The application's datasource is connected to a reachable database
- Operator has access to the Schematic UI URL in a browser
- The target table exists in the database
- The datasource user has DROP privilege on the target table

## Main Success Scenario

1. Operator navigates to the table list in the Schematic UI.
2. System displays the list of tables.
3. Operator selects the Drop Table action for the target table.
4. System displays a confirmation dialog naming the table to be dropped and warning that the action is irreversible.
5. Operator confirms the drop operation.
6. System drops the table from the database.
7. System removes the table from the displayed table list and shows a success message.

## Alternative Flows

### A1: Operator Cancels Confirmation

**Trigger:** Operator dismisses or cancels the confirmation dialog (step 5)
**Flow:**

1. System closes the confirmation dialog without executing any operation.
2. Table list remains unchanged.
3. Use case ends.

### A2: Drop Fails Due to Foreign Key Constraint

**Trigger:** The target table is referenced by a foreign key in another table (step 6)
**Flow:**

1. System displays an error message indicating the table cannot be dropped because of an existing dependency.
2. Operator reviews the dependency and resolves it before retrying.
3. Use case ends.

### A3: Drop Fails Due to Insufficient Privileges

**Trigger:** The datasource user does not have DROP privilege on the target table (step 6)
**Flow:**

1. System displays an error message indicating insufficient database privileges.
2. Operator contacts a database administrator to obtain the required privilege.
3. Use case ends.

### A4: Database Connection Lost During Drop

**Trigger:** The datasource cannot reach the database when the drop is executed (step 6)
**Flow:**

1. System displays an error message indicating the database is unreachable.
2. Table state is indeterminate; operator verifies the table status after connectivity is restored.
3. Use case ends.

### A5: Table No Longer Exists at Execution Time

**Trigger:** The target table has already been removed by another process between step 3 and step 6
**Flow:**

1. System displays an error message indicating the table was not found.
2. System refreshes the table list to reflect the current database state.
3. Use case ends.

## Postconditions

### Success Postconditions

- The table and all its data are permanently removed from the database
- The table no longer appears in the Schematic table list
- All indexes and constraints belonging to the dropped table are also removed

### Failure Postconditions

- The table remains in the database unchanged
- System displays an error message indicating the cause of failure
- No other tables or data are affected

## Business Rules

### BR-001: Mandatory Confirmation

Every Drop Table operation must display an explicit confirmation dialog naming the target table before executing; the operation cannot proceed without operator confirmation.

### BR-002: Irreversibility Warning

The confirmation dialog must clearly communicate that the operation is irreversible and that all data in the table will be permanently lost.

### BR-003: Datasource Permissions Apply

The drop executes under the privileges of the host application's configured datasource user; Schematic does not elevate or bypass database permissions.
