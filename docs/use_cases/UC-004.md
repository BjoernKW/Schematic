# Use Case: Execute SQL Query

## Overview

**Use Case ID:** UC-004
**Use Case Name:** Execute SQL Query
**Primary Actor:** Developer
**Goal:** Execute a custom SQL query through the UI to investigate data and troubleshoot issues without a separate database client
**Status:** Draft

## Preconditions

- Schematic is embedded in the running Spring Boot application (UC-001 completed)
- The application's datasource is connected to a reachable database
- Developer has access to the Schematic UI URL in a browser

## Main Success Scenario

1. Developer navigates to the SQL query view in the Schematic UI.
2. System displays a SQL input area.
3. Developer enters a SQL query.
4. Developer submits the query.
5. System executes the query against the connected database using the application's datasource.
6. System displays the query results in a tabular format with column headers.

## Alternative Flows

### A1: Query Returns No Rows

**Trigger:** The executed query produces an empty result set (step 5)
**Flow:**

1. System displays the column headers with no data rows and an appropriate empty-state message.
2. Use case ends.

### A2: SQL Syntax Error

**Trigger:** The submitted query contains invalid SQL syntax (step 5)
**Flow:**

1. System displays an error message describing the syntax problem.
2. Developer corrects the query in the input area.
3. Use case continues at step 4.

### A3: Query Execution Error

**Trigger:** A valid query fails at runtime (e.g., missing table, permission denied) (step 5)
**Flow:**

1. System displays the database error message returned by the datasource.
2. Developer reviews the error, adjusts the query, and resubmits.
3. Use case continues at step 4.

### A4: Query Returns a Large Result Set

**Trigger:** The result set exceeds the display row limit (step 6)
**Flow:**

1. System displays the first N rows and indicates that the result was truncated.
2. Developer refines the query to narrow the result set if needed.
3. Use case ends.

### A5: Empty Query Submitted

**Trigger:** Developer submits a blank or whitespace-only query (step 4)
**Flow:**

1. System does not execute any query.
2. System displays a validation message prompting the developer to enter a query.
3. Use case continues at step 3.

### A6: Database Connection Unavailable

**Trigger:** The datasource cannot reach the database when the query is submitted (step 5)
**Flow:**

1. System displays an error message indicating the database is unreachable.
2. Developer checks connectivity and retries.
3. Use case ends.

## Postconditions

### Success Postconditions

- Query results are displayed to the developer
- The database reflects any changes made by DML or DDL statements in the query

### Failure Postconditions

- No query results are displayed
- System shows an error message indicating the cause
- The database is unchanged if the query did not execute

## Business Rules

### BR-001: Datasource Permissions Apply

Queries execute under the permissions of the host application's configured datasource user; Schematic does not grant or restrict database privileges beyond those already held by that user.

### BR-002: Result Row Limit

The number of rows displayed is capped to protect memory in the host application process; queries that return more rows than the cap produce a truncated result with a visible indicator.

### BR-003: In-Process Execution

All query execution occurs within the host Spring Boot process using the existing datasource; no additional network connections or ports are opened.
