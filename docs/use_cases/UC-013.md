# Use Case: Use Query History

## Overview

**Use Case ID:** UC-013
**Use Case Name:** Use Query History
**Primary Actor:** Developer
**Goal:** Access previously executed SQL queries from a dropdown and re-run or refine them without retyping
**Status:** Done

## Preconditions

- Schematic is embedded in the running Spring Boot application (UC-001 completed)
- Developer has access to the Schematic UI in a browser that supports the Web Storage API
- Developer has executed at least one SQL query during a previous session (UC-004 completed at least once)

## Main Success Scenario

1. Developer navigates to the Schematic UI and focuses on the SQL query input.
2. System displays a query history dropdown populated with previously executed queries, most recent first.
3. Developer selects an entry from the dropdown.
4. System copies the selected SQL text into the query input field.
5. Developer reviews the query, optionally modifies it, and submits.
6. System executes the query and displays the results (UC-004).
7. System prepends the newly executed query to the history and persists the updated list in browser local storage.

## Alternative Flows

### A1: No History Entries Exist

**Trigger:** Developer opens the history dropdown but no queries have been stored yet (step 2)
**Flow:**

1. System displays an empty dropdown with an informational placeholder (e.g. "No history yet").
2. Use case ends.

### A2: Developer Executes a New Query Without Using History

**Trigger:** Developer types a query directly and submits without selecting from history (step 5)
**Flow:**

1. System executes the query and displays the results (UC-004).
2. System prepends the query to the history and persists the updated list in browser local storage.
3. Use case ends.

### A3: Duplicate Query Submitted

**Trigger:** The submitted query text is identical to the most recent history entry (step 7)
**Flow:**

1. System does not add a duplicate entry; the existing entry moves to the top of the list.
2. Use case ends.

### A4: History Reaches Maximum Capacity

**Trigger:** The number of stored history entries reaches the configured maximum (step 7)
**Flow:**

1. System removes the oldest entry from the list to make room.
2. System prepends the new query and persists the updated list.
3. Use case ends.

### A5: Developer Clears Query History

**Trigger:** Developer activates the "Clear history" control
**Flow:**

1. System removes all entries from browser local storage for the history key.
2. System displays an empty history dropdown.
3. Use case ends.

### A6: Browser Local Storage Unavailable

**Trigger:** The browser does not support or has disabled the Web Storage API (step 7)
**Flow:**

1. System executes the query and displays the results normally.
2. System does not attempt to persist the history entry.
3. The history dropdown is not shown for the remainder of the session.
4. Use case ends.

## Postconditions

### Success Postconditions

- The executed query appears at the top of the history dropdown in subsequent interactions
- The query text is retrievable across browser sessions as long as local storage is not cleared
- No data in the database is modified by the history feature itself

### Failure Postconditions

- If local storage is unavailable, queries execute normally but history is not persisted
- No partial or corrupted entries are written to local storage

## Business Rules

### BR-001: Client-Side Storage Only

Query history is stored exclusively in browser local storage. No history data is sent to or persisted by the server. Each browser and device maintains its own independent history.

### BR-002: Maximum History Size

The history list is capped at 50 entries. When the cap is reached, the oldest entry is removed before a new one is added.

### BR-003: Deduplication

If the submitted query text is identical to the most recent entry, no duplicate is added. The entry's position is refreshed to the top of the list instead.

### BR-004: Most-Recent-First Ordering

History entries are always displayed with the most recently executed query at the top of the dropdown.

### BR-005: History Scope

History is scoped to the browser origin. Queries executed in one browser or device are not visible in another.

### BR-006: Graceful Degradation

If the Web Storage API is unavailable or throws an exception, the SQL query execution feature continues to work normally. The history dropdown is silently suppressed for that session.
