# Use Case: Add Library to Project

## Overview

**Use Case ID:** UC-001
**Use Case Name:** Add Library to Project
**Primary Actor:** Developer
**Goal:** Embed the Schematic database UI into an existing Spring Boot application by adding a single Maven dependency
**Status:** Draft

## Preconditions

- Developer has an existing Spring Boot 3.x or 4.x application with a configured datasource
- The application uses Maven as its build tool
- The application has network access to Maven Central (or a local mirror)
- Java 21 is the runtime

## Main Success Scenario

1. Developer opens the project's `pom.xml`.
2. Developer adds the Schematic dependency to the `<dependencies>` section with the latest version from Maven Central.
3. Developer rebuilds the application.
4. System downloads the Schematic JAR and its transitive dependencies from Maven Central.
5. System detects the Schematic auto-configuration at startup and wires all required beans using the existing datasource.
6. Developer starts the application.
7. System exposes the Schematic UI at `/schematic/tables`.
8. Developer opens a browser and navigates to `/schematic/tables`.
9. System displays the database schema browser.

## Alternative Flows

### A1: Library Already Present

**Trigger:** Schematic dependency already exists in `pom.xml` (step 2)
**Flow:**

1. Developer updates the version number to the desired release.
2. Use case continues at step 3.

### A2: Dependency Resolution Fails

**Trigger:** Maven cannot resolve the artifact from any configured repository (step 4)
**Flow:**

1. System reports a build failure with a dependency resolution error.
2. Developer verifies the version number and repository configuration.
3. Use case continues at step 3.

### A3: Datasource Not Configured

**Trigger:** No datasource bean is present in the host application at startup (step 5)
**Flow:**

1. System fails to start and logs a missing datasource error.
2. Developer adds datasource configuration to `application.yml`.
3. Use case continues at step 6.

### A4: Port Conflict on Default Path

**Trigger:** The host application already exposes a route at `/schematic` (step 7)
**Flow:**

1. Developer configures `schematic.path` in `application.yml` to an unused path.
2. Use case continues at step 6 with the UI available at the configured path (see UC-009).

### A5: Custom UI Mount Path Configured

**Trigger:** Developer sets `schematic.path` in `application.yml` before starting the application (step 6)
**Flow:**

1. System exposes the UI at the configured path instead of the default `/schematic/tables`.
2. Use case continues at step 8 with the custom path.

## Postconditions

### Success Postconditions

- The Schematic UI is accessible in the running application at `/<schematic.path>/tables`
- The library uses the host application's existing datasource without any additional configuration
- No existing routes, beans, or application behaviour are affected

### Failure Postconditions

- The host application either fails to start or starts without the Schematic UI
- No changes to the host application's existing functionality occur
- Build or startup error messages indicate the root cause

## Business Rules

### BR-001: Single Dependency Activation

Adding the Maven dependency is the only required step to activate Schematic; no annotations, imports, or bean declarations in the host application are required.

### BR-002: Datasource Reuse

Schematic must use the host application's existing configured datasource and must not require a dedicated secondary datasource.

### BR-003: No Port Isolation

Schematic runs entirely within the host Spring Boot process and does not open any additional network ports.

### BR-004: No Transitive Conflicts

The Schematic JAR must not introduce transitive dependency conflicts with a standard Spring Boot 3.x or 4.x application.
