# Use Case: Configure Access Control

## Overview

**Use Case ID:** UC-012
**Use Case Name:** Configure Access Control
**Primary Actor:** Developer
**Goal:** Register a bean in the host application that restricts which tables are visible and which operations are permitted so that Schematic can be safely exposed in non-development environments
**Status:** Draft

## Preconditions

- Schematic is added as a Maven dependency in the host application (UC-001 completed)
- Developer has identified tables that should be hidden or operations that should be restricted
- Developer has write access to the host application's source code

## Main Success Scenario

1. Developer creates a class in the host application that implements the `SchematicTableFilter` interface.
2. Developer annotates the class as a Spring bean (e.g. `@Component` or `@Bean`).
3. Developer implements the interface methods to specify which tables are visible and which operations are permitted for each table.
4. Developer starts or restarts the application.
5. System detects the `SchematicTableFilter` bean in the application context at startup.
6. Developer navigates to the Schematic table listing.
7. System applies the filter and displays only tables that the bean permits as visible.
8. Developer navigates to the ER diagram view.
9. System generates the diagram using only the visible tables.
10. Developer attempts a destructive operation on a visible table.
11. System checks whether the operation is permitted by the bean for that table and either presents the confirmation dialog or rejects the request.

## Alternative Flows

### A1: No Bean Registered

**Trigger:** No `SchematicTableFilter` bean is present in the application context (step 5)
**Flow:**

1. System applies no filtering; all tables are visible and all operations are permitted.
2. Use case ends.

### A2: Bean Hides All Tables

**Trigger:** The bean's visibility logic returns false for every table in the schema (step 7)
**Flow:**

1. System displays an empty table listing with an appropriate informational message.
2. System generates an empty ER diagram.
3. Use case ends.

### A3: Table Visible but Operation Restricted

**Trigger:** The bean permits a table to be visible but disallows a specific operation on it (step 11)
**Flow:**

1. System displays the table in the listing and allows row preview.
2. System hides or disables the restricted operation control for that table.
3. If the developer attempts the restricted operation directly via URL, the system rejects the request and returns an error response.
4. Use case ends.

### A4: Bean Updated After Startup

**Trigger:** Developer modifies the `SchematicTableFilter` implementation after the application is running
**Flow:**

1. The updated filtering logic does not take effect while the application is running.
2. Developer restarts the application.
3. Use case continues at step 5.

### A5: Bean Throws an Exception During Evaluation

**Trigger:** The `SchematicTableFilter` bean throws an unchecked exception when queried for a table (step 7 or step 11)
**Flow:**

1. System treats the affected table as not visible and all its operations as not permitted (fail-closed).
2. System logs the exception with the table name for diagnostic purposes.
3. Use case continues for the remaining tables.

## Postconditions

### Success Postconditions

- Only tables permitted by the `SchematicTableFilter` bean appear in the listing, row preview, and ER diagram
- Restricted operations are not accessible through the Schematic UI for the affected tables
- No data in the database is modified by registering the bean

### Failure Postconditions

- If no bean is registered, Schematic behaves as if no filter is in effect
- If the bean throws exceptions, affected tables are hidden and their operations are blocked; other tables are unaffected
- The host application's existing routes and Spring context are not affected

## Business Rules

### BR-001: Default Permissive Behaviour

If no `SchematicTableFilter` bean is registered in the application context, all tables are visible and all operations (DROP TABLE, TRUNCATE TABLE) are permitted. This preserves the zero-configuration principle for read access.

### BR-002: Visibility Implies Row Preview Access

If a table is marked as not visible, row preview and the table detail view are also inaccessible for that table, regardless of any operation-level settings.

### BR-003: Filter Scope

The bean's visibility rules apply consistently across the table listing, row preview, and ER diagram. A table that is not visible does not appear in any of these views.

### BR-004: SQL Execution Is Not Filtered

Ad-hoc SQL queries entered in the query input are not subject to the `SchematicTableFilter`. The filter controls the UI-level table browsing and operation controls; raw SQL access is governed by the host application's own security configuration.

### BR-005: Fail-Closed on Exception

If the `SchematicTableFilter` bean throws an exception when evaluated for a specific table, the system treats that table as not visible and its operations as not permitted, preventing accidental exposure due to implementation errors in the bean.

### BR-006: Restart Required

Changes to the `SchematicTableFilter` bean take effect only after the application is restarted. Live reload of the filtering configuration is not supported.
