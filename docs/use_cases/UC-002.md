# Use Case: Browse Database Schema

## Overview

**Use Case ID:** UC-002
**Use Case Name:** Browse Database Schema
**Primary Actor:** Developer
**Goal:** View all tables and their columns in the connected database to understand the schema without a separate database client
**Status:** Done

## Preconditions

- Schematic is embedded in the running Spring Boot application (UC-001 completed)
- The application's datasource is connected to a reachable database
- Developer has access to the Schematic UI URL in a browser

## Main Success Scenario

1. Developer navigates to the Schematic UI at `/<schematic.path>/tables`.
2. System queries the database for all user-accessible tables via INFORMATION_SCHEMA.
3. System displays the list of tables with their names.
4. Developer selects a table from the list.
5. System retrieves the column definitions for the selected table via INFORMATION_SCHEMA.
6. System displays the column names, data types, and nullability for the selected table.

## Alternative Flows

### A1: No Tables Found

**Trigger:** The connected database contains no user-accessible tables (step 2)
**Flow:**

1. System displays an empty table list with an appropriate message.
2. Use case ends.

### A2: Database Connection Unavailable

**Trigger:** The datasource cannot reach the database when the page is loaded (step 2)
**Flow:**

1. System displays an error message indicating the database is unreachable.
2. Developer checks the datasource configuration and network connectivity.
3. Use case ends.

### A3: Sort and Filter Applied

**Trigger:** Developer sorts or filters the table list (step 3) — extends UC-010
**Flow:**

1. Developer enters a filter term or selects a sort order.
2. System updates the displayed table list to match the filter and sort criteria without a full page reload.
3. Use case continues at step 4.

### A4: Large Number of Tables

**Trigger:** The schema contains a very large number of tables, causing a slow initial render (step 2)
**Flow:**

1. System loads and displays whatever results are available within the performance budget.
2. Use case continues at step 3.

## Postconditions

### Success Postconditions

- Developer can see the full list of tables in the connected database
- Developer can see the column definitions (name, type, nullability) for any selected table
- No data in the database is modified

### Failure Postconditions

- The table list or column details are not displayed
- System shows an error message indicating the cause
- No data in the database is modified

## Business Rules

### BR-001: INFORMATION_SCHEMA Reliance

Schema introspection must rely solely on standard INFORMATION_SCHEMA views so that it works with any JDBC-compatible database.

### BR-002: Read-Only Operation

Browsing the schema is a read-only operation; no database structures or data are altered.

### BR-003: Page Load Performance

The table list page must complete its initial render within 2 seconds on a developer workstation connected to a local database.
